Fiber Churn Intelligence — Technical Design Page

This page is the technical design reference for the churn prediction solution: what the solution does, how the API is structured, what input values are required, why those values matter, and how the response is interpreted for business use.

1. High-Level Design (HLD)

Purpose

The project predicts whether a telecom fiber customer is likely to churn using behavior, pricing, usage, and service quality signals. It supports retention campaign decisions by exposing threshold-based business logic.

Core components

  • Dataset generation: creates realistic customer behavior patterns in fiber_customers_realistic.csv.
  • Training pipeline: trains and compares models in train_model.py, saving the best model and feature columns.
  • Prediction API: serves inference through FastAPI in app/main.py.
  • Threshold tuning: stores campaign-oriented cutoffs in threshold_tuning.json.

System flow

  1. Generate realistic customer records.
  2. Engineer features such as tenure, pricing, outages, complaints, and contract signals.
  3. Train and select the best model.
  4. Persist the model and feature schema.
  5. Expose /predict for real-time scoring and /threshold-tuning for threshold metadata.

2. Low-Level Design (LLD)

API surface

  • GET /health — checks service health and model availability.
  • POST /predict — scores a customer and returns churn risk.
  • GET /threshold-tuning — returns campaign threshold profiles.
  • GET /feature-importance — returns feature contribution values from the trained model.

Model inputs

The prediction service accepts one JSON object with the following mandatory fields. All values are required because the trained model uses them to build the feature vector.

5. Query Parameters and Business Logic

  • campaign (optional, default: balanced) — selects a threshold profile for business use.
  • threshold (optional) — overrides the campaign threshold when a custom cutoff is needed.

Available campaign profiles are defined in threshold_tuning.json:

  • retention_campaign — aggressive, good for outreach campaigns.
  • balanced — default middle-ground threshold.
  • conservative — stricter, fewer but higher-confidence alerts.

6. Response Shape

Interpretation: a higher probability means the customer is more likely to churn. The selected threshold decides how aggressive the retention campaign should be.

8. How churn is trained and what basis is used

The training label is not guessed manually. The dataset generator creates a churn score from business signals and converts that score into a probability using a logistic function, then samples a binary label from that probability.

Churn score formula used during dataset generation

churn_score =
  0.12 * min(outages, 6)
+ 0.10 * min(complaints, 6)
+ 0.08 * min(supportCalls, 6)
+ 0.14 * min(latePayments, 5)
+ 0.15 * competitorAvailable
+ 0.10 * (tenureMonths < 6)
+ 0.08 * (monthlyPrice > 1200)
+ 0.12 * recentPlanChange
+ 0.10 * contractRenewalDue
+ 0.10 * (avgMonthlyUsageGb < 80)
+ 0.03 * (monthlyContract == 1 and tenureMonths < 12)
+ normal_noise(0, 0.15)

That score is then converted to a churn probability with the sigmoid function:

p(churn) = 1 / (1 + e^(-churn_score))

The final label is generated as churn = 1 when a random draw is below this probability, otherwise 0. In other words, the training target is based on realistic telecom risk drivers rather than arbitrary labels.

How the model learns

  • The model sees the customer features listed in the input contract and learns the relationship between those features and the generated churn label.
  • Training uses stratified train/test splits and compares models with RandomizedSearchCV.
  • The evaluation uses F1 score, ROC-AUC, and a combined score of 0.7 * F1 + 0.3 * ROC-AUC to select the best model.

How the final churn decision is made at prediction time

probability = model.predict_proba(customer)[0][1]
churn_prediction = 1 if probability >= threshold else 0

The threshold is selected from campaign profiles such as retention_campaign, balanced, and conservative. These thresholds are tuned using precision, recall, and F1 score over candidate cutoffs from 0.1 to 0.9.

9. Why this design is useful

  • It gives a clear API contract for other services to call the prediction model safely.
  • It separates data science logic from business logic through threshold tuning profiles.
  • It supports usage in real retention campaigns, not only model experimentation.
  • It provides a simple way to integrate with applications, dashboards, or CRM workflows.

Outcome: better targeting, reduced churn, and faster operational adoption.
